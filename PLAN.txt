------------------------------------------------------------------------
1. THE CORE IDEA (one sentence)
------------------------------------------------------------------------
An Omegle-style, ELO-ranked battle game where two users are given the
same prompt and must "LARP" a small/mundane achievement into an
impressive-sounding LinkedIn-style flex — the better, more *believable*
and domain-authentic LARP wins, as scored by an AI judge.

Logline: "Omegle for LinkedIn flexing." / "Competitive corporate LARPing,
ELO-ranked like chess.com."


------------------------------------------------------------------------
2. WHY THIS IDEA (the pitch / the soul)
------------------------------------------------------------------------
- LinkedIn is notoriously full of performative flexing — larping tech,
  larping business. This game satirizes that culture by making it a sport.
- It's meta and timely: many businesses now AI-screen applicants'
  LinkedIn profiles, so having an AI *judge* the LARP is on-theme.
- The skill is real: good LARPing isn't rambling nonsense. It's using
  REAL domain knowledge to make a mundane thing sound impressive but
  still plausible.
- It accidentally teaches a real skill: how to communicate and frame
  your own experience/profile as more than it literally is.
- Emotional hook needs zero explanation — a judge gets it in one sentence
  and can re-pitch it to a friend. That's the whole game on Devpost.

Reference points for the vibe:
- chess.com ELO / ranked ladder
- "mog battles" (Omegle-style 1v1 where two people compete on a metric)


------------------------------------------------------------------------
3. HOW THE GAME WORKS (mechanics)
------------------------------------------------------------------------
3.1 The prompt
- Each battle gives a prompt / persona to LARP around.
- Prompts are things a person might genuinely have done, at varying
  prestige levels:
    * Low/mundane:   "Clash Royale enthusiast"
    * Mid:           "Participated in a science olympiad (didn't win)"
    * High/prestige: "You won ISEF"
- The challenge scales: mundane prompts test creativity; prestigious
  prompts test RESTRAINT (larp it big without going off-the-rails cringe).

3.2 The LARP (what the user writes)
- The user writes ONE paragraph of LARP.
- Good LARP = authentic domain references, not empty buzzwords.
- Worked example — prompt "Clash Royale enthusiast":
    "I learned to mitigate my resources efficiently [elixir management],
     strategized and deployed agents effectively at my disposal, and
     built a coordinated team of units to produce the cleanest possible
     output under pressure."
  → This works because it maps REAL game knowledge (elixir economy, card
    cycling, win conditions) onto corporate language. That's the skill.

3.3 The judging (this is the innovation surface — see Con #2)
- NOT a single AI judge. A PANEL of 3 AI persona judges scores each LARP
  (full panel defined in Section 4.5).
- Each judge owns a rubric axis and has its own voice + verdict.
- Scores shown WITH per-judge one-line justifications (transparent,
  not a black-box number).

3.4 The battle + ranking
- Two LARPs on the same prompt are compared head-to-head.
- Winner determined by total score across categories.
- ELO updates for both players (chess.com-style ladder).
- Leaderboard shows top LARPers.


------------------------------------------------------------------------
4. THE SCORING RUBRIC (the heart of the product)
------------------------------------------------------------------------
Proposed categories (this is where the "not just a wrapper" lives):

  A. PLAUSIBILITY / DOMAIN AUTHENTICITY
     Does the flex reference REAL knowledge of the topic?
     (Clash Royale elixir management = high; vague nonsense = low)

  B. RESTRAINT / BELIEVABILITY
     Did they oversell into cringe, or keep it just-believable?
     (Winning ISEF gracefully > "I single-handedly cured cancer")

  C. BUZZWORD DENSITY  (satirical axis)
     Deliberately double-edged — too many buzzwords can LOSE points.
     This bakes the joke into the scoring itself.

  D. DETECTABILITY  (meta / on-theme axis)
     "Would an AI recruiter screener flag this as fake?"
     Rewards LARP that survives the same AI screening real businesses use.

- Show the breakdown per category with a short justification each.
- THAT visible mechanism is what upgrades this from "ChatGPT wrapper"
  to "clever system." Spend tonight's prompt-engineering budget HERE.

4.5 THE JUDGING PANEL — 3 AI PERSONAS  (LOCKED: "The Hiring Panel" / Trio A)
--------------------------------------------------------------------------
Instead of one AI judge, a panel of 3 personas judges each LARP. The
personas are the entertainment — three characters arguing over your LARP
is funnier and more memorable than a number, and it justifies the
multi-axis rubric because each judge OWNS one axis with its own
opposing win condition (so they actually disagree).

  JUDGE 1 — THE BURNT-OUT RECRUITER
     Owns: Detectability (axis D)
     Has seen 10,000 profiles, sniffs out fakes instantly.
     Win condition: wants to CATCH you lying. Hard to fool.

  JUDGE 2 — THE BUZZWORD VC
     Owns: Buzzword appeal (axis C)
     Dazzled by "synergy," "10x," "disrupt."
     Win condition: wants to be IMPRESSED. Easily fooled.

  JUDGE 3 — THE GEN-Z INTERN
     Owns: Restraint / Believability (axis B)
     Allergic to cringe, roasts anything try-hard.
     Win condition: wants you to NOT embarrass yourself.

  (Axis A, Domain Authenticity, is folded into the Recruiter's read of
  the LARP, or can be split out later. Keep to 3 judges for the MVP.)

  NOTE: mechanically this is the Hiring Panel. It can later be re-skinned
  in LinkedIn-archetype flavor (Thought Leader / HR Voice / Grindset Bro)
  for extra satire without changing the underlying logic.

4.6 PANEL IMPLEMENTATION NOTES  (so it stays cheap + distinct)
--------------------------------------------------------------------------
- SINGLE structured LLM call, not 3 calls. One prompt returns all 3
  personas' verdicts as a JSON array; parse into three "speakers." Same
  cost/latency as one judge. (See Con #6.)
- Make judges STRUCTURALLY opposed via their win conditions above, not
  just tonally — opposing incentives force real disagreement. Tone alone
  makes them collapse into one bland voice. (See Con #7.)
- CROSS-TALK (high-leverage "wow"): prompt judges to briefly reference
  each other — e.g. "The Intern's right, this is cringe, but I'd still
  hire them." Makes it feel like a real panel, not 3 separate scores.
- MOOD/BIAS (optional replay hook): a judge can get a per-round bias
  (e.g. VC gives bonus for buzzwords THIS round), shifting the optimal
  LARP each battle. One line in the prompt.
- REVEAL THEATRE: show judges ONE AT A TIME with score + one-liner —
  turns LLM latency into a show. Split verdicts (2 love it, 1 flags it)
  are the best demo moments; consensus is boring.
- AGGREGATION RULE (decide now, not at 1am): average the 3 scores into
  the battle result that feeds ELO. Simple and defensible.

5. DESIGN DECISION LOCKED: USER-WRITES (not AI-generates)
------------------------------------------------------------------------
Two possible versions were considered:
  (a) User WRITES their own LARP  ← CHOSEN
  (b) AI generates LARPs, user picks/rates

Chosen (a) because it IS the game that was pitched — "your ability to
LARP." It's funnier, has real replay value, and makes the ELO actually
mean something. Version (b) is faster to build but removes the soul and
produces a generic product.

7. CONS + SOLUTIONS (the risks, and how to beat each)
------------------------------------------------------------------------

CON #1 — REAL-TIME 1v1 MULTIPLAYER IS A TRAP  (BIGGEST RISK)
  Problem: Omegle-style live matchmaking = WebSockets, matchmaking queue,
  lobby/sync state, disconnect handling, and "what if nobody's online at
  1 AM to demo against." That's the whole night, and it demos badly
  because you need a second live human.
  SOLUTION: Build ASYNC battles for the MVP. User submits a LARP, gets
  instantly matched against a STORED previous submission (or an AI
  "house" opponent), scored, ELO updates. Feels single-player to run but
  is structurally still PvP. Demos solo. Live multiplayer → future slide.

CON #2 — "AI-JUDGED" IS WHERE THIN IDEAS GET EXPOSED
  Problem: If the score is just "ask the model to rate 1–10," judges see
  through it instantly and it reads as a wrapper.
  SOLUTION: The RUBRIC is the innovation. Use the 4 legible, clever
  categories (Section 4) and SHOW per-category justifications. Spend the
  prompt-engineering effort here, not on extra features.

CON #3 — LLM LATENCY WILL WRECK THE "BATTLE" FEEL
  Problem: Two scoring calls + justifications can take 5–15s. A battle
  hanging on a spinner feels broken.
  SOLUTION: Stream the scoring, OR make the wait THEATRICAL — animate the
  "AI judges deliberating… detecting buzzwords… scoring plausibility…"
  reveal with scores counting up. Turn latency into showmanship; it
  demos better than instant.

CON #4 — COLD-START / EMPTY-ARENA PROBLEM
  Problem: Async battles need opponents to already exist; an empty
  leaderboard feels dead.
  SOLUTION: Seed the DB with 20–30 pre-generated LARPs across prompts
  before demo (a ~15-min script). Populates the leaderboard so the ELO
  feels real.

CON #5 — CONTENT-SAFETY EDGE  (minor but real)
  Problem: Open-ended generation + real institution/person names could
  produce genuinely deceptive/defamatory output (e.g., LARPing as a named
  real doctor giving real medical advice).
  SOLUTION: Keep prompts to fictionalized/generic flexes, add a visible
  "this is satire" banner, and don't allow LARPing as a named real person
  giving real professional advice. Low effort, don't overthink it.

CON #6 — 3 JUDGES MULTIPLY COST + LATENCY  (introduced by the panel)
  Problem: 3 personas = naively 3x the calls, straight into the latency
  problem (Con #3).
  SOLUTION: ONE structured LLM call that returns all 3 verdicts as JSON;
  parse into three speakers. Same cost/latency as a single judge. Do NOT
  make 3 parallel calls for the MVP — a good prompt fakes independence
  for free.

CON #7 — PERSONAS COLLAPSE INTO THE SAME VOICE
  Problem: LLMs default to a bland average; three "distinct" judges can
  come out sounding identical, killing the whole point.
  SOLUTION: Make them STRUCTURALLY opposed — give each a different WIN
  CONDITION (Recruiter catches fakes, VC wants to be dazzled, Intern
  avoids cringe), not just a different tone. Opposing incentives force
  real disagreement.

11. EXECUTION PLAN — CLAUDE CODE BUILD SEQUENCE  (zero-rework ordered)
------------------------------------------------------------------------
GOAL: build the COMPLETE project end to end with Claude Code, step by
step. This is NOT ordered for "safe to submit anytime" — it is ordered
for AGENTIC BUILD EFFICIENCY: define contracts once, build logic before
UI, mock the LLM early, and never rebuild a layer because a lower layer
changed under it.

WHY THIS ORDER (the 4 efficiency rules it follows):
  1. CONTRACTS FIRST. Lock the data shapes (LARP, Verdict, Battle,
     Player) before any code touches them, so nothing downstream gets
     rewritten when a field changes.
  2. LOGIC BEFORE UI. Build + test scoring/ELO as pure functions the UI
     later just renders. UI wired to real logic from its first render.
  3. MOCK THE EXPENSIVE THING EARLY. Stub the LLM call behind an
     interface returning canned verdicts, so you build 80% of the app
     with zero latency / zero API spend, then swap in the real call once.
  4. ONE VERTICAL SEAM AT A TIME. Each step ends with something you can
     run and eyeball in the terminal or browser before moving on.

HOW TO DRIVE CLAUDE CODE:
  - One step = one focused prompt. Don't ask it to "build the app" —
    give it the current step's spec + exit criteria from below.
  - After each step, RUN it and confirm the exit criteria yourself
    before starting the next. Commit at every step boundary.
  - Keep this PLAN.txt in the repo so Claude Code has the full spec as
    context (Sections 3, 4, 4.5, 4.6 are its source of truth).

RECOMMENDED STACK (pick before Step 0, tell Claude Code once):
  - Vite + React, single page. TypeScript recommended — the type
    definitions ARE the contracts, and Claude Code stays far more
    consistent when the shapes are typed.
  - LLM behind ONE server-side function (Vercel/Netlify function or tiny
    Express route). API key in env, never client-side.
  - Storage: a single storage module with a clean interface (get/save
    players, battles). Back it with localStorage first; the interface
    means you can swap in a real DB later without touching callers.
